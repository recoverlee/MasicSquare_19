# Level 1 — EPIC: Business Goal

**Epic:** Invariant(불변조건) 기반 사고 훈련 시스템 구축

**목적**  
4×4 Magic Square 문제를 매개로 다음을 달성한다.

- Invariant(불변조건) 중심의 설계·사고 훈련
- Dual-Track TDD(요구/테스트와 구현의 이중 궤적) 적용 훈련
- 입력/출력 계약(Contract)의 명시와 검증 훈련
- 설계 → 테스트 → 구현 → 리팩토링 순서의 습관화

**성공 기준**

| 구분 | 기준 |
|------|------|
| Domain logic | 도메인 로직 테스트 커버리지 **95% 이상** |
| 입력 | 입력에 대한 **계약 테스트 100% 통과** |
| 품질 | **하드코딩·매직 넘버** 없이 일관되게 구현 |
| 추적성 | Invariant(또는 요구)에서 테스트까지 **명시적 추적 가능** |

# Level 2 — User Journey (개요)

### Persona

- **소프트웨어 개발 학습자** — 작은 시스템을 스스로 설계·검증하려는 단계
- **TDD 훈련 중** — RED → GREEN → REFACTOR를 요구·회귀의 수단으로 쓴다
- **Clean Architecture 이해 중** — UI·응용·도메인·(필요 시) I/O 경계를 구분하려는 단계

### Journey

1. **문제 인식** — “구현한다”에 몰지 말고, **관찰·질문**으로 맥락·**성공**의 경계를 먼저 쓴다.  
2. **계약 정의** — 입력 스키마, 출력 스키마, **예외 정책**  
3. **Domain 분리** — 예: `BlankFinder`, `MissingNumberFinder`, `MagicSquareValidator`, `Solver` (역할은 한 클래스에 뭉쟈 금지)  
4. **Dual-Track** — UI RED / Logic RED **병렬** → GREEN 최소 → REFACTOR **통합**  
5. **회귀 보호** — 엣지, 입력 오류, 조합/탐색 **실패** 케이스

# Level 3 — User Stories

### Story 1 — 입력 검증

**As a** 학습자 **I want** 4×4 입력이 **경계에서** 검증되기를 **So that** 잘못된 데이터가 Domain으로 넘어가지 않는다

**AC:** 4×4가 아님(예외) · 빈칸(0) **2개** 아님(예외) · 0 제외 **중복**(예외) · 1~16 **범위** 위반(예외)

### Story 2 — 빈칸 탐색

**As a** 학습자 **I want** `0`의 **좌표**를 찾기를 **So that** 조합 시도가 **결정론**적으로 가능하다

**AC:** **row-major** 순서 · **정확히 2개** 좌표(또는 이에 맞는 표현)

### Story 3 — 누락 숫자 탐색

**As a** 학습자 **I want** 1~16에서 **빠진 2수**를 구하기를 **So that** Solver **후보**가 고정된다

**AC:** 1~16(각 1회) 전제 시 **누락 2개** · **오름차순** (예: `[a,b]`, `a ≤ b`)

### Story 4 — 마방진 판정

**AC:** 모든 **행** 합 동일 · 모든 **열** 합 동일(행과 동일) · (문제가 요구하면) **두 대각** 동일(교재의 **34**와 일치)

### Story 5 — 두 조합(첫·역) 시도

**AC:** **small → first blank**(row-major 첫) 우선 **시도** · **실패 시** 합의된 **reverse(역시도)** **두 번째 시도** · 계약: **“정답 배열 길이 6”** (예: `[r1,c1,v1, r2,c2,v2]`, 좌표 **1-index** 등**문서와 동일**)

> Story 5·Gherkin의 “역시도 **만** 성공”은 **픽스처 A ≠ B** 인 **별도** 숫자 격자가 있어야 논리가 맞다.

# Level 4 — Technical 시나리오 (Gherkin)

```gherkin
# language: ko
Feature: 4x4 마방진 완성

  Invariant에 맞는 로직을 검증하며 TDD를 연습하는 개발자로서
  0=빈칸 2곳인 4x4 격자를 마방진(매직 합 34)으로 완성한 결과를 얻고 싶다

  Background:
    Given 4x4 정수 격자가 주어지고
    And 0은 빈칸을 뜻하며
    And 0 셀은 정확히 2개이고
    And 0이 아닌 셀의 값은 1 이상 16 이하이고
    And 0이 아닌 셀끼리는 중복이 없고
    And 완성 4x4(일반) 마방진의 기대 합(매직 합)은 34다

  Scenario: 작은 수 → 큰 수(첫 시도)로 완성되는 경우
    Given 다음 격자(픽스처 A)가 주어졌다:
      | 16 |  2 |  3 | 13 |
      |  5 | 11 | 10 |  8 |
      |  9 |  7 |  0 | 12 |
      |  4 | 14 | 15 |  0 |
    When 시스템이 row-major로 빈칸 2곳을 찾고
    And 1~16에서 누락된 수 2개를 구한 뒤
    And 작은 수를 row-major **첫** 빈칸에, 큰 수를 **둘째** 빈칸에 반영한 결과로 격자를 완성하면
    Then 완성 격자는 모든 행/열/두 대각선이 각 34이고
    And 계약에 따라 “길이 6” 결과(예: [r1, c1, v1, r2, c2, v2])가 반환되고
    And r, c는 1~4(1-index)다

  Scenario: 역시도(두 번째 시도)에서만 마방진이 되는 경우
    Given 픽스처 A와 **다른** B가
    And (작→첫, 큰→둘)은 34에 실패하고
    And (큰→첫, 작→둘)(역시도)은 34에 **성공**하도록 **정의**됐다
    When “첫 시도”는 실패, “역시도”는 성공이면
    Then 반환은 **역시도**에 맞는 유효 해만, 완성 격자는 **34 Invariant**를 만족한다

  Scenario: 빈칸 개수가 잘못됨
    Given 0(빈칸) 개수가 2가 아닌 4x4
    When 입력 유효성 검증을 수행하면
    Then 오류

  Scenario: 0이 아닌 셀에 중복
    Given 0이 아닌 셀에 중복 값
    When 입력 유효성 검증
    Then 오류

  Scenario: 값이 범위 밖
    Given 0이 아닌 셀에 1~16 밖
    When 입력 유효성 검증
    Then 오류
```

# Level 5 — 시나리오 검증 및 정리 (체크리스트)

## 4레벨 일관성

- [ ] Epic 성공 기준이 **Journey·AC·Gherkin**에 반영됨  
- [ ] **Pain**이 “계약·SRP·Invariant·테스트” **다음 행동**으로 연결됨  
- [ ] Story **AC**가 Gherkin/자동화 테스트와 **1:1** 또는 **추적 테이블**이 있음  
- [ ] `Given-When-Then`이 **입력·행위·기대(또는 예외)** 를 빼먹지 않음

## Edge (본 실습: 로컬 도메인)

- [ ] Happy path(또는 계약이 정의한) **긍정** 시나리오  
- [ ] 입력 계약: **4×4 / 빈칸 2 / 중복·범위**  
- [ ] 경계: **1, 16**, `0=빈`, **row-major**  
- [ ] “역시도”용 **픽스처 B** (A와 **다른** 격자) **정의** 여부  
- (범용: 네트워크·권한·QR·오프라인DB 등) — **4×4만**이면 **N/A**
